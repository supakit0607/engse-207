# การวิเคราะห์โปรเจกต์ (Project Analysis)

เอกสารนี้ให้การวิเคราะห์แอปพลิเคชันกระดานงาน (Task Board) และวิธีที่สถาปัตยกรรม N-Tier ของมันตอบสนองหลักการทางวิศวกรรมซอฟต์แวร์ (software engineering principles) ที่สำคัญ

## ความต้องการเชิงฟังก์ชัน (Functional Requirements)

เป้าหมายหลักของโปรเจกต์นี้คือการสร้างแอปพลิเคชันกระดานงานที่เรียบง่ายแต่ใช้งานได้จริง แอปพลิเคชันต้องรองรับคุณสมบัติดังต่อไปนี้:

-   **การจัดการงาน (Task Management):** ผู้ใช้ควรสามารถสร้าง (create), ดู (view), อัปเดต (update), และลบ (delete) งานได้
-   **สถานะงาน (Task Status):** งานควรมีสถานะที่แตกต่างกัน (เช่น "ต้องทำ", "กำลังดำเนินการ", "เสร็จสิ้น") และผู้ใช้ควรสามารถเปลี่ยนสถานะของงานได้
-   **API:** แอปพลิเคชันต้องเปิดเผย RESTful API สำหรับฟังก์ชันการทำงานทั้งหมด
-   **ส่วนติดต่อผู้ใช้ (User Interface):** ควรมีส่วนติดต่อผู้ใช้พื้นฐานเพื่อโต้ตอบกับ API

## ความต้องการที่ไม่ใช่เชิงฟังก์ชัน (Non-Functional Requirements)

-   **การบำรุงรักษา (Maintainability):** โค้ดเบส (codebase) ควรง่ายต่อการทำความเข้าใจและแก้ไข
-   **ความสามารถในการปรับขนาด (Scalability):** สถาปัตยกรรม (architecture) ควรสามารถรองรับจำนวนผู้ใช้และงานที่เพิ่มขึ้นได้
-   **ความสามารถในการทดสอบ (Testability):** ควรสามารถทดสอบส่วนต่างๆ ของแอปพลิเคชันได้อย่างอิสระ (in isolation)
-   **การแยกความรับผิดชอบ (Separation of Concerns):** โค้ดของแอปพลิเคชันควรได้รับการจัดระเบียบตามความรับผิดชอบของมัน

## สถาปัตยกรรม N-Tier ตอบสนองความต้องการอย่างไร

สถาปัตยกรรม N-Tier ถูกเลือกสำหรับโปรเจกต์นี้เนื่องจากสามารถตอบสนองความต้องการที่ไม่ใช่เชิงฟังก์ชันได้อย่างมีประสิทธิภาพ ซึ่งจะช่วยอำนวยความสะดวกในการใช้งานความต้องการเชิงฟังก์ชัน

-   **การบำรุงรักษา (Maintainability):** โดยการแยกแอปพลิเคชันออกเป็นเลเยอร์ (layers) (การนำเสนอ, ธุรกิจ, การเข้าถึงข้อมูล) สถาปัตยกรรม N-Tier ทำให้โค้ดมีความเป็นระเบียบและง่ายต่อการบำรุงรักษามากขึ้น นักพัฒนาสามารถทำงานในเลเยอร์เดียวโดยไม่จำเป็นต้องเข้าใจรายละเอียดของเลเยอร์อื่นๆ ตัวอย่างเช่น นักพัฒนาส่วนหน้าสามารถทำงานในไดเรกทอรี `public/` โดยไม่จำเป็นต้องรู้โครงสร้างฐานข้อมูล (database schema)

-   **ความสามารถในการปรับขนาด (Scalability):** สถาปัตยกรรม N-Tier ช่วยให้แต่ละเทียร์ (tier) สามารถปรับขนาดได้อย่างอิสระ (independently) ตัวอย่างเช่น หากแอปพลิเคชันประสบปัญหาการใช้งานสูง เราสามารถปรับขนาดเว็บเซิร์ฟเวอร์ (เทียร์การนำเสนอ) และเซิร์ฟเวอร์แอปพลิเคชัน (เทียร์ธุรกิจ) แยกกันได้ ในทำนองเดียวกัน หากฐานข้อมูลเป็นคอขวด (bottleneck) เราสามารถปรับขนาดเทียร์ข้อมูล (Data Tier) ได้

-   **ความสามารถในการทดสอบ (Testability):** การแยกเลเยอร์ทำให้แอปพลิเคชันง่ายต่อการทดสอบ เราสามารถทดสอบตรรกะทางธุรกิจในไดเรกทอรี `services/` โดยไม่จำเป็นต้องมีส่วนติดต่อผู้ใช้หรือฐานข้อมูล เราสามารถจำลอง (mock) repositories เพื่อทดสอบบริการแยกกันได้ (in isolation)

-   **การแยกความรับผิดชอบ (Separation of Concerns):** นี่คือหลักการสำคัญของสถาปัตยกรรม N-Tier แต่ละเลเยอร์มีความรับผิดชอบเฉพาะตัว ซึ่งนำไปสู่โค้ดเบสที่เป็นโมดูลาร์ (modular) และจัดระเบียบได้ดีขึ้น
    -   **เทียร์การนำเสนอ (Presentation Tier)** เพียงแค่เกี่ยวข้องกับการแสดงข้อมูลและการจัดการอินพุตของผู้ใช้
    -   **เทียร์ธุรกิจ (Business Tier)** เพียงแค่เกี่ยวข้องกับตรรกะทางธุรกิจ
    -   **เทียร์การเข้าถึงข้อมูล (Data Access Tier)** เพียงแค่เกี่ยวข้องกับการเข้าถึงฐานข้อมูล

การแยกความรับผิดชอบที่ชัดเจนนี้ทำให้แอปพลิเคชันมีความยืดหยุ่น (flexible) มากขึ้นและง่ายต่อการพัฒนาและบำรุงรักษาในระยะยาว
